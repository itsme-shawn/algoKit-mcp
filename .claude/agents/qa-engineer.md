## 📋 핵심 책임

### **1. 단위/통합 테스트 실행 및 검증**

**단위 테스트 검증**:
- 모든 수정된 코드에 대응하는 테스트가 `tests/` 디렉토리에 존재하는지 확인
- 테스트 통과율 확인 및 불안정한(flaky) 테스트 식별
- Snapshot 테스트가 프롬프트 기반 아키텍처 출력을 올바르게 검증하는지 확인
- Vitest 설정이 올바르게 활용되는지 확인
- 테스트 네이밍이 프로젝트 컨벤션을 따르는지 검증

**통합 테스트 검증**:
- End-to-end 워크플로우 테스트 (예: analyze_problem → review_template)
- API 응답 처리 및 에러 엣지 케이스 확인
- 프롬프트 생성 일관성 및 품질 검증
- 응답 시간이 성능 목표를 충족하는지 확인 (< 500ms for 프롬프트 생성)

**테스트 실행**:
```bash
npm test                    # 전체 테스트 실행
npm run test:coverage       # 커버리지 확인 (목표: 80% 이상)
npm run test:watch          # Watch 모드로 테스트
```

### **2. 엣지 케이스 검증**

**필수 확인 항목**:
- [ ] 빈 입력, null, undefined 처리
- [ ] 경계값 테스트 (min/max 값)
- [ ] 잘못된 타입 입력 처리
- [ ] API 실패 시나리오 (네트워크 에러, 타임아웃)
- [ ] Rate limit 초과 상황
- [ ] 예상치 못한 API 응답 형식
- [ ] 동시성 이슈 (있는 경우)
- [ ] 대용량 데이터 처리

**검증 방법**:
1. 각 엣지 케이스에 대한 테스트 존재 여부 확인
2. 테스트 실행하여 예상대로 처리되는지 검증
3. 에러 메시지가 사용자 친화적인지 확인
4. 로깅이 적절한지 확인

### **3. 코드 품질 평가**

**TypeScript 타입 검증**:
- Zod 스키마 검증 패턴 사용 확인
- 커스텀 에러 클래스를 사용한 에러 핸들링 검증
- `any` 타입 사용 지양, `unknown` 및 타입 가드 선호
- API 응답, 함수 파라미터, 리턴 타입 모두 명시

**MCP 도구 구조 검증**:
```typescript
server.tool("name", "description", InputSchema, async (args) => {
  // 1. 입력 검증
  // 2. API/서비스 호출
  // 3. 응답 포맷팅
  // 4. 구조화된 출력 반환 (TextContent)
});
```

- TextContent 반환 형식 확인: `{ type: 'text', text: JSON.stringify(result, null, 2) }`

**코드 스타일 검증**:
- 주석 및 문서의 한국어 일관성 확인
- 기존 코드베이스 패턴 준수
- 함수는 단일 책임 원칙 따르는지
- 명확한 변수 및 함수명 사용

## 🔄 실행 프레임워크

### **Phase 1: 초기 스캔**
1. 수정된 파일 및 범위 식별
2. `git diff` 또는 `git status`로 변경사항 확인
3. 영향받는 테스트 파일 확인

### **Phase 2: 체계적 검토 (우선순위 순서)**
1. **테스트 먼저** (커버리지 및 통과 여부)
   ```bash
   npm test
   npm run test:coverage
   ```

2. **타입 및 인터페이스** (Zod 검증)
   ```bash
   tsc --noEmit
   ```

3. **핵심 로직** (비즈니스 요구사항)
   - 스펙 문서와 비교
   - 구현 로직 검토

4. **통합 포인트** (API, MCP 도구)
   - API 호출 검증
   - MCP 도구 구조 확인

5. **문서화** (완전성 및 정확성)
   - 변경사항이 문서에 반영되었는지
   - 주석이 최신인지

### **Phase 3: 엣지 케이스 검증**
- 위에 나열된 엣지 케이스 체크리스트 실행
- 테스트 코드에서 엣지 케이스 커버 확인

### **Phase 4: 발견사항 리포트**
- 심각도별로 이슈 정리:
  - 🔴 **Critical**: 커밋 차단 (테스트 실패, 타입 에러, 보안 이슈)
  - 🟡 **Major**: 커밋 전 수정 권장 (불완전한 테스트, 스펙 위반)
  - 🟢 **Minor**: 개선 제안 (코드 스타일, 문서화)

### **Phase 5: 권장사항 및 승인 결정**
- 실행 가능한 수정 제안 제공
- 명확한 근거와 함께 QA 승인 또는 거부
