# AlgoKit : Software Requirement Specification

> 코테 (BOJ) 학습 도우미 MCP Server 의 기능, 비기능적 요구사항, 제한 사항, 목표 등을 상세히 기술한 문서

## 1. solved.ac API 조사

### API 기본 정보
- **Base URL**: `https://solved.ac/api/v3`
- **인증 (Authentication)**: API 키 불필요 (공개 API)
- **요청 제한 (Rate Limits)**: 공식 문서화되지 않음, 합리적인 throttling 사용 권장
- **CORS**: 브라우저에서 호출 불가 (서버 측에서만 사용 가능)

### 주요 엔드포인트

#### 1.1 문제 검색
```
GET /search/problem
```
**쿼리 파라미터**:
- `query` (string): 검색어 (문제 제목, 번호, 또는 태그 키워드)
- `page` (number): 페이지 번호 (기본값: 1)
- `sort` (string): 정렬 기준 (예: "level", "id", "average_try")
- `direction` (string): "asc" 또는 "desc"

**응답 구조**:
```json
{
  "count": 22,
  "items": [
    {
      "problemId": 1000,
      "titleKo": "A+B",
      "titles": [...],
      "level": 1,
      "acceptedUserCount": 363116,
      "averageTries": 2.6,
      "tags": [...],
      "isSolvable": true
    }
  ]
}
```

**난이도 필터링**: 태그 기반 필터링 또는 레벨 범위 사용
- 레벨: 1-30 (1=Bronze V, 6=Bronze I, 11=Silver I, 16=Gold I, 21=Platinum I, 26=Diamond I, 30=Ruby I)

#### 1.2 문제 상세 정보 조회
```
GET /problem/show?problemId={problemId}
```
**응답**: 다음을 포함한 전체 문제 메타데이터:
- 문제 ID, 제목 (한글/영문)
- 레벨 (티어)
- 표시명과 별칭이 포함된 태그
- 통계 (해결한 사용자 수, 평균 시도 횟수)

#### 1.3 태그 검색
```
GET /search/tag?query={query}
```
**응답**: 쿼리와 일치하는 알고리즘 태그 목록
- 태그 키 (예: "dp", "greedy")
- 여러 언어의 표시명
- 각 태그의 문제 수

### API 특성
- JSON 응답 반환
- 인증 불필요
- 태그 시스템은 키 사용 (예: "dp", "graphs", "greedy")
- 티어/레벨은 난이도를 나타내는 1-30 숫자
- 문제는 한글과 영문 제목 모두 제공

## 2. MCP 도구 설계

### 도구 1: `search_problems`
**목적**: 필터를 사용한 BOJ 문제 검색

**입력 스키마**:
```typescript
{
  query?: string;           // 검색어 (제목, 번호, 태그)
  level_min?: number;       // 최소 레벨 (1-30)
  level_max?: number;       // 최대 레벨 (1-30)
  tag?: string;             // 알고리즘 태그 키 (예: "dp", "greedy")
  sort?: "level" | "id" | "average_try";
  direction?: "asc" | "desc";
  page?: number;            // 기본값: 1
}
```

**출력**: 필수 정보(ID, 제목, 레벨, 태그)가 포함된 문제 목록

### 도구 2: `get_problem`
**목적**: 특정 문제의 상세 정보 조회

**입력 스키마**:
```typescript
{
  problem_id: number;       // BOJ 문제 번호
}
```

**출력**: 전체 문제 상세 정보 (제목, 티어, 태그, 통계, BOJ 링크)

### 도구 3: `search_tags`
**목적**: 키워드로 알고리즘 태그 찾기

**입력 스키마**:
```typescript
{
  query: string;            // 태그 검색 쿼리 (예: "dynamic")
}
```

**출력**: 이름과 키가 포함된 일치하는 태그 목록

### 도구 4: `get_hint`
**목적**: 문제에 대한 단계적 힌트 생성

**입력 스키마**:
```typescript
{
  problem_id: number;       // BOJ 문제 번호
  hint_level: number;       // 1-3 (기본 → 상세 → 해결 접근법)
  user_context?: string;    // 선택사항: 사용자의 현재 이해도
}
```

**출력**: 문제 태그와 난이도 기반의 맥락 인식 힌트

**힌트 생성 전략**:
- 레벨 1: 문제 패턴 인식 (어떤 유형의 알고리즘인지)
- 레벨 2: 완전한 해법 없이 핵심 통찰이나 접근법
- 레벨 3: 상세한 알고리즘 단계 (실제 코드는 제외)

### 도구 5: `create_review`
**목적**: 문제 해결 후 마크다운 복습 문서 생성

**입력 스키마**:
```typescript
{
  problem_id: number;       // BOJ 문제 번호
  solution_approach: string; // 사용자의 접근법 설명
  time_complexity?: string; // 선택사항
  space_complexity?: string; // 선택사항
  key_insights?: string;    // 선택사항
  difficulties?: string;    // 선택사항
}
```

**출력**: 구조화된 복습이 포함된 마크다운 파일 내용

**복습 템플릿**:
```markdown
# [문제 번호] 문제 제목 (티어)

## 문제 링크
https://www.acmicpc.net/problem/{id}

## 태그
- 태그1, 태그2, 태그3

## 난이도
티어: Gold III (레벨 13)

## 해결 접근법
{사용자_제공_접근법}

## 시간 복잡도
O(...)

## 공간 복잡도
O(...)

## 핵심 인사이트
- 인사이트 1
- 인사이트 2

## 어려웠던 점 & 배운 점
{도전적이었던_내용}

## 관련 문제
- 태그 기반 유사 문제 제안

---
*해결 날짜: {date}*
*통계: {해결한_사용자_수}명 해결, 평균 {평균_시도} 시도*
```

## 3. 시스템 아키텍처

### 3.1 컴포넌트 구조

```
mcp-server/
├── src/
│   ├── index.ts                 # MCP 서버 진입점
│   ├── api/
│   │   ├── solvedac-client.ts   # API 클라이언트 래퍼
│   │   └── types.ts             # API 응답 타입
│   ├── tools/
│   │   ├── search-problems.ts   # 도구: search_problems
│   │   ├── get-problem.ts       # 도구: get_problem
│   │   ├── search-tags.ts       # 도구: search_tags
│   │   ├── get-hint.ts          # 도구: get_hint (LLM 기반)
│   │   └── create-review.ts     # 도구: create_review
│   ├── services/
│   │   ├── hint-generator.ts    # 힌트 생성 로직
│   │   └── review-generator.ts  # 복습 템플릿 생성기
│   └── utils/
│       ├── tier-converter.ts    # 레벨 숫자 ↔ 티어 이름
│       └── cache.ts             # 선택사항: 응답 캐싱
├── tests/
│   ├── api/
│   │   └── solvedac-client.test.ts
│   └── tools/
│       ├── search-problems.test.ts
│       └── get-hint.test.ts
└── package.json
```

### 3.2 데이터 흐름

```
사용자 요청 → MCP 도구 → solved.ac API 클라이언트 → API 응답
                ↓
         서비스 계층 (힌트/복습 생성)
                ↓
         포맷된 응답 → 사용자
```

### 3.3 주요 설계 결정사항

**힌트 생성**:
- **접근법**: LLM 기반 (프롬프트를 통한 Claude 사용)
- **이유**: 문제 힌트는 문제 유형, 난이도, 사용자의 현재 지식에 대한 맥락 이해가 필요
- **방법**: 도구가 문제 메타데이터 수신 → 문제 태그/레벨로 프롬프트 구성 → 힌트 생성

**복습 생성**:
- **접근법**: 템플릿 기반, 선택적 LLM 향상
- **이유**: 복습은 구조화된 형식을 가지지만, 관련 문제 제안에 LLM 활용 가능
- **방법**: 사용자 입력 + 문제 메타데이터로 템플릿 채우기 + 유사 문제 제안

**캐싱**:
- **선택사항**: 문제 상세 정보에 대한 간단한 인메모리 캐시 구현
- **이유**: 자주 접근하는 문제에 대한 API 호출 감소
- **TTL**: 1시간 (문제 메타데이터는 거의 변경되지 않음)

**에러 처리**:
- API 실패를 우아하게 처리
- API 다운 시 대체 응답 제공
- API 호출 전 사용자 입력 검증

## 4. 태스크 분해

### 개발자 태스크

#### Phase 1: 기반 구축 (우선순위: 높음)
- **Task 1.1**: 프로젝트 구조 및 TypeScript 설정
  - MCP SDK로 npm 프로젝트 초기화
  - tsconfig.json 구성
  - 빌드 스크립트 설정

- **Task 1.2**: solved.ac API 클라이언트 구현
  - HTTP 클라이언트(fetch/axios)로 `solvedac-client.ts` 생성
  - API 응답용 TypeScript 인터페이스 정의
  - 에러 처리 및 요청 재시도 로직 구현
  - 선택적 응답 캐싱 추가

- **Task 1.3**: 티어/레벨 유틸리티 구현
  - 티어 변환기 생성 (레벨 숫자 ↔ 티어 이름)
  - 티어 뱃지 표시 헬퍼 추가

- **Task 2.1**: `search_problems` 도구 구현
  - Zod로 입출력 스키마 정의
  - 쿼리 파라미터로 API 클라이언트 호출
  - 결과 포맷 및 필터링
  - 페이지네이션 처리

- **Task 2.2**: `get_problem` 도구 구현
  - ID로 문제 상세 정보 조회
  - BOJ 링크로 응답 포맷
  - 모든 메타데이터 포함 (태그, 티어, 통계)

- **Task 2.3**: `search_tags` 도구 구현
  - 태그 검색 엔드포인트 쿼리

#### Phase 3: 고급 기능 (우선순위: 중간)

  - 3단계 힌트 프롬프트 템플릿 설계
  - LLM(Claude API) 통합
  - 태그 기반 맥락 인식 힌트 생성
  - 다양한 문제 유형으로 테스트

- **Task 3.2**: `get_hint` 도구 구현
  - 문제 메타데이터 조회
  - 힌트 생성 서비스 호출
  - 단계적 힌트 반환

- **Task 3.3**: 복습 생성 서비스 구현
  - 마크다운 복습 템플릿 생성

  - 사용자 입력을 복습 형식으로 포맷

- **Task 3.4**: `create_review` 도구 구현
  - 사용자 입력 수집
  - 구조화된 마크다운 생성
  - 선택적으로 파일로 저장

### QA 태스크

#### 테스트 전략
- **단위 테스트 (Unit Tests)**: 개별 도구 및 서비스 테스트
- **통합 테스트 (Integration Tests)**: 실제 엔드포인트로 API 클라이언트 테스트
- **E2E 테스트**: 전체 MCP 도구 워크플로우 테스트

#### 테스트 케이스

**TC1: 문제 검색**
- 쿼리어로 검색
- 티어 범위로 필터링
- 알고리즘 태그로 필터링
- 페이지네이션 정상 작동
- 빈 결과 우아하게 처리

**TC2: 문제 상세 정보 조회**
- 유효한 문제 ID는 올바른 데이터 반환
- 유효하지 않은 문제 ID는 에러 반환
- 응답에 모든 필수 필드 포함

**TC3: 태그 검색**
- 검색이 일치하는 태그 반환
- 다국어 지원 작동
- 태그 키가 올바름

**TC4: 힌트 생성**
- 레벨 1 힌트는 충분히 일반적
- 레벨 2 힌트는 유용한 통찰 제공
- 레벨 3 힌트는 완전한 해법을 제공하지 않음
- 힌트가 문제 난이도 및 태그와 일치

**TC5: 복습 생성**
- 템플릿이 적절히 채워짐
- 사용자 입력이 올바르게 포맷됨
- 관련 문제가 관련성 있음
- 마크다운 문법이 유효함

**TC6: 에러 처리**
- API 실패가 포착됨
- 유효하지 않은 입력은 명확한 에러 메시지 반환
- 네트워크 타임아웃 처리

**TC7: 성능**
- 캐싱이 API 호출 감소
- 응답 시간이 허용 가능 (대부분 요청 <2초)
